# Sports Data Integrations — 5-Phase Execution Plan

## Scope
Backend-only implementation for sports data ingestion (no odds ingestion). Providers:
- API-Tennis (tennis)
- Goalserve (horse racing)
- BetsAPI/B365API (greyhound, with horse/tennis backup capability)

Data objective:
- Fixtures
- Live status
- Results
- Participants
- Start times (UTC)

Out of scope:
- Odds fetching/odds markets from external providers
- Frontend/UI work

---

## Phase 1 — Foundation, Schema, and Contracts
### Goal
Create the base integration architecture and unified data model used by all providers.

### Tasks
1. Define unified domain contract for normalized events:
   - `provider`
   - `provider_event_id`
   - `sport`
   - `competition_name`
   - `status`
   - `start_time_utc`
   - `participants`
   - `result`
   - `raw`
2. Add migration + schema for `sports_events` table:
   - UUID PK
   - unique index on `[:provider, :provider_event_id]`
   - filter indexes for `sport`, `status`, `start_time_utc`
3. Create ingestion service/context:
   - `upsert_events/1` with conflict target `[:provider, :provider_event_id]`
   - conflict update strategy for mutable fields (`status`, `start_time_utc`, `participants`, `result`, `raw`, timestamps)
4. Add shared provider utilities:
   - safe datetime parsing helpers
   - status mapping helper conventions
   - defensive list handling helpers (`List.wrap/1` policy)
   - common error return shape (`{:error, reason}`)
5. Add provider runtime config keys (env-driven only):
   - `API_TENNIS_KEY`
   - `GOALSERVE_KEY`
   - `BETSAPI_TOKEN`

### Deliverables
- DB schema + Ecto schema ready
- Unified ingest pipeline ready
- Shared utilities and config contracts ready

### Exit Criteria
- Any normalized event map can be inserted/upserted successfully
- No hardcoded secrets
- All times stored as UTC

---

## Phase 2 — Provider Clients + Normalizers (REST)
### Goal
Implement production-safe HTTP clients and normalization modules for all 3 providers.

### Tasks
1. API-Tennis module:
   - endpoints: event types, tournaments, fixtures, livescore
   - enforce `timezone=UTC` on requests
   - validate envelope `success == 1`
   - normalize participants and result payload
2. Goalserve module:
   - build key-in-path URL format
   - always request `?json=1`
   - parse nested tournament/race/horse using `List.wrap/1`
   - parse `DD.MM.YYYY HH:MM` in provider timezone and convert to UTC
3. BetsAPI module:
   - primary host `api.b365api.com`, fallback `api.betsapi.com`
   - endpoints: upcoming, inplay, ended, event view, search
   - handle `results`/`result` variations and pagination
   - parse Unix timestamps to UTC
4. Provider-specific status mappers:
   - API-Tennis status and `event_live` mapping
   - Goalserve race status mapping
   - BetsAPI `time_status` mapping
5. Error resilience:
   - graceful handling for non-200, malformed payload, API-level errors
   - standardized logging metadata (`provider`, `endpoint`, `status`, `reason`)

### Deliverables
- 3 provider client modules
- 3 normalizers outputting unified event shape
- robust error handling and fallback logic

### Exit Criteria
- Each provider can fetch and normalize sample fixture/live/result payloads
- No unhandled crash path for bad provider payloads

---

## Phase 3 — Polling, Scheduling, and Rate-Limit Safety
### Goal
Automate ingestion with Oban workers and provider-safe polling strategies.

### Tasks
1. Create Oban workers:
   - `TennisFetchWorker`
   - `HorseRacingFetchWorker`
   - `GreyhoundFetchWorker`
2. Implement provider cadence strategy:
   - tennis: live ~60s / daily fixture horizon refresh
   - horse racing: race-day frequent polling + slower pre-race refresh
   - greyhound: live every minute + upcoming/ended windows
3. Add rate-limit controls:
   - API-Tennis: minimum delay between calls
   - Goalserve: avoid hammering; enforce minimum intervals
   - BetsAPI: <= default quota cadence, parse rate-limit headers
4. 429 handling and backoff:
   - BetsAPI snooze until reset timestamp
   - exponential backoff/retry where appropriate
5. Pagination + batch ingest:
   - fetch all pages where pager exists
   - chunk upserts to avoid large transaction pressure

### Deliverables
- Oban Cron scheduling config
- stable workers with retry/snooze behavior
- ingestion telemetry logs for each run

### Exit Criteria
- Workers can run continuously without rate-limit violations
- Incremental updates correctly refresh event status/results

---

## Phase 4 — Realtime and Webhook Extensions
### Goal
Add low-latency pathways for live updates and complete operational behavior.

### Tasks
1. API-Tennis WebSocket (Business plan path):
   - WebSockex process + supervisor
   - auto reconnect with exponential backoff
   - parse push events and normalize/upsert immediately
   - REST fallback when socket unhealthy > threshold
2. Goalserve webhook receiver:
   - `POST /webhooks/goalserve`
   - verify payload shape, normalize, ingest
   - safe response handling and logging
3. Idempotency and duplication controls:
   - dedupe by provider/provider_event_id
   - guard against out-of-order updates where possible
4. Operational observability:
   - structured logs for ingest success/fail counts
   - provider-level health indicators (last success, last error, lag)

### Deliverables
- supervised websocket pipeline
- Goalserve webhook endpoint
- realtime ingestion path integrated with existing upsert flow

### Exit Criteria
- Live updates continue during transient disconnects
- Webhook and websocket events are idempotently stored

---

## Phase 5 — Compliance, Hardening, and Go-Live Readiness
### Goal
Finalize production readiness, legal checks, and runbook-level stability.

### Tasks
1. Legal/commercial readiness checklist:
   - Goalserve commercial rights confirmation for betting use
   - BetsAPI greyhound limitation documented (MVP-only)
   - production replacement plan for licensed greyhound feed
2. Security and config hardening:
   - ensure all secrets are env-based
   - redact secrets in logs
   - provider error logs free of sensitive tokens
3. Data quality safeguards:
   - validate required normalized fields before upsert
   - quarantine/reject malformed events with diagnostics
4. Backfill and recovery procedures:
   - historical sync jobs (date-ranged)
   - replay strategy for missed polling windows
5. Acceptance checklist and handover docs:
   - endpoint matrix by provider
   - rate-limit policy table
   - failure-mode runbook (429, 403 whitelist, malformed payload, provider outage)

### Deliverables
- production hardening completed
- go-live readiness checklist signed off
- operational runbook and support playbook

### Exit Criteria
- integration stack operates reliably under normal and degraded provider conditions
- legal and operational blockers are explicitly resolved/documented

---

## Cross-Phase Implementation Standards
1. Never raise on provider failures in production paths; return `{:error, reason}`.
2. Keep all timestamps in UTC at persistence boundary.
3. Ignore external odds fields entirely.
4. Maintain strict provider-specific parsing rules (especially Goalserve list wrapping).
5. Keep each provider integration isolated behind its own module/context for maintainability.

---

## Recommended Build Order (Execution Sequence)
1. Phase 1 first (schema + ingestion contract).
2. Phase 2 provider clients in this order:
   - API-Tennis
   - Goalserve
   - BetsAPI
3. Phase 3 worker automation and rate-limit safety.
4. Phase 4 realtime channels (websocket + webhook).
5. Phase 5 hardening + compliance before production rollout.
