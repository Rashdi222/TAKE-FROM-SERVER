# Provider API Management — Execution Plan (Backend)

## Objective
Build a super-admin controlled API Management system so request budgets are not exhausted unexpectedly.
Admin must be able to configure per-provider limits, pause/resume providers, and monitor usage in real time.

---

## Scope
- Backend only (Phoenix + Ecto + PostgreSQL)
- Super-admin protected endpoints only
- Real-time-ish usage observability via API (UI can poll)

Out of scope for this phase:
- Frontend implementation
- External alerting channels (Slack/Email) unless explicitly added later

---

## Architecture Summary
1. Add DB-backed provider control settings.
2. Add rolling usage counters per minute/hour/day.
3. Add a limiter service that gates provider requests.
4. Integrate limiter checks into provider workers before each API call.
5. Expose management and reporting endpoints for super admin.
6. Audit all settings changes.

---

## Phase A — Data Model & Migrations

### A1. Create `provider_api_controls` table
Fields:
- `id` (uuid PK)
- `provider_key` (string, unique, indexed)
- `enabled` (boolean, default true)
- `poll_interval_seconds` (integer, nullable)
- `min_delay_ms_between_calls` (integer, nullable)
- `max_requests_per_minute` (integer, nullable)
- `max_requests_per_hour` (integer, nullable)
- `max_requests_per_day` (integer, nullable)
- `auto_pause_at_percent` (integer, nullable, e.g. 80-100)
- `paused_until` (utc_datetime, nullable)
- `notes` (text, nullable)
- `inserted_at`, `updated_at` (utc_datetime)

Indexes/constraints:
- unique index on `provider_key`
- check constraints for non-negative numeric fields
- check `auto_pause_at_percent` between 1 and 100 when not null

### A2. Create `provider_api_usage_windows` table
Fields:
- `id` (uuid PK)
- `provider_key` (string, indexed)
- `window_type` (enum-like string: `minute|hour|day`)
- `window_start_utc` (utc_datetime)
- `request_count` (integer, default 0)
- `success_count` (integer, default 0)
- `failure_count` (integer, default 0)
- `blocked_count` (integer, default 0)
- `inserted_at`, `updated_at` (utc_datetime)

Indexes/constraints:
- unique index on `provider_key, window_type, window_start_utc`
- index on `window_type, window_start_utc`
- non-negative counter checks

### A3. Create `provider_api_events` table
Fields:
- `id` (uuid PK)
- `provider_key` (string, indexed)
- `event_type` (string)
  - `request_ok`, `request_failed`, `request_blocked`, `auto_paused`, `manual_paused`, `resumed`, `settings_changed`
- `payload` (jsonb)
- `actor_id` (uuid nullable, FK users.id for admin actions)
- `inserted_at` (utc_datetime)

Indexes:
- `provider_key, inserted_at desc`
- `event_type, inserted_at desc`

### A4. Seed default control rows
For each active provider key in system, ensure default row exists.

## Phase C — Rate Limiter Engine

### C1. Add service `Back.ApiManagement.RateLimiter`
Required functions:
- `allow_request?(provider_key)`
- `record_success(provider_key)`
- `record_failure(provider_key)`
- `record_blocked(provider_key, reason)`

### C2. Decision flow for `allow_request?`
1. Load provider control row.
2. If disabled -> block.
3. If paused_until in future -> block.
4. Resolve current minute/hour/day windows.
5. Compare counters against configured limits.
6. If over threshold -> block and maybe auto-pause.
7. If allowed -> increment request counter atomically and return `:ok`.

### C3. Atomicity
- Use `Repo.transaction` + upsert/increment to avoid race conditions.
- Ensure parallel workers do not bypass limits.

### C4. Auto-pause logic
- If usage reaches configured threshold percent, set `paused_until` and emit `auto_paused` event.

## Phase D — Worker & Provider Integration

### D1. Integrate limiter with sports workers
Files:
- `TennisFetchWorker`
- `HorseRacingFetchWorker`
- `GreyhoundFetchWorker`

Before each outbound request:
- call `allow_request?(provider_key)`
- if blocked: skip call, log blocked event, continue gracefully

After request:
- success -> `record_success`
- failure -> `record_failure`

### D2. Integrate min-delay and poll interval controls
- Respect `min_delay_ms_between_calls`
- Respect provider `poll_interval_seconds` in scheduling logic

### D3. Fallback behavior
- If primary provider blocked/paused, optionally try configured fallback provider if enabled.

## Phase E — Super Admin API Endpoints

### E1. Controller
Add `BackWeb.ApiManagementController` (super-admin scope).

### E2. Endpoints
1. `GET /api/super-admin/api-management/providers`
- List controls + current usage snapshot

2. `GET /api/super-admin/api-management/providers/:provider_key`
- Provider control detail + usage + recent events

3. `PUT /api/super-admin/api-management/providers/:provider_key`
- Update limits/settings

4. `POST /api/super-admin/api-management/providers/:provider_key/pause`
- Pause immediately, optional until timestamp

5. `POST /api/super-admin/api-management/providers/:provider_key/resume`
- Resume provider

6. `GET /api/super-admin/api-management/usage`
- Aggregated usage (minute/hour/day) with remaining quota

7. `GET /api/super-admin/api-management/events`
- Events feed with filters and pagination

8. `POST /api/super-admin/api-management/providers/:provider_key/reset-usage`
- Manual reset counters (audited)

### E3. Response contract
Success:
```json
{
  "data": {"...": "..."},
  "meta": {"timestamp": "...", "pagination": {"...": "..."}}
}
```
Error:
```json
{
  "error": "message",
  "errors": {"field": ["msg"]}
}
```

## Acceptance Criteria
1. Super admin can configure limits per provider without deploy.
2. Providers stop calling APIs when limits are hit.
3. Usage counters are accurate under concurrent worker load.
4. Pause/resume works immediately.
5. Admin can view usage + event history via API.
6. Existing betting and provider functionality remains intact.

## Implementation Note
This module should be developed in an isolated namespace (`Back.ApiManagement`) and integrated into workers via explicit calls only, to minimize regression risk.
