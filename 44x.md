# SportsMonks Cricket Batch Polling Upgrade Plan

## Goal
Reduce unnecessary SportsMonks request volume while keeping live cricket state accurate enough for odds generation, repricing, and public live boards.

This plan keeps the current architecture:
- SportsMonks remains the match-state source for cricket
- 1xBet remains the source/reference odds source where mapped
- LangGraph continues repricing from live cricket context

It changes how SportsMonks data is fetched so the platform can support more enabled leagues and more live matches without polling every feed too frequently.

## Current Problem
The current code path still leans on frequent generic live polling and feed-level refreshes. That works for small scale, but it becomes expensive and noisy when:
- many cricket leagues are enabled
- many live matches overlap
- the platform is also running live AI repricing and provider-reference recovery

The real issue is not just request count. It is also uneven freshness:
- some matches get polled too often
- some matches do not need full-detail fetches every cycle
- the system does not clearly separate discovery polling from per-match detail refresh

## Relevant Current Code
- `back/lib/back/providers/sportmonks.ex`
  - live entrypoint currently uses `/livescores`
  - fixture entrypoint uses `/fixtures`
  - fixture enrichment can call `GET /fixtures/{id}` when team context is incomplete
- `back/lib/back/workers/match_fetcher.ex`
  - generic live sync loop runs every 4 seconds
  - competition-level live sync runs every 15 seconds
- `back/lib/back/live/cricket_sportmonks_consumer.ex`
  - converts incoming live updates into match state changes
- `ai_engine/cricket/*`
  - depends on frequent but stable match-state updates

## Official API Notes
Based on SportMonks cricket API documentation:
- live state entrypoint is `/livescores`
- fixture detail entrypoint is `/fixtures/{ID}`
- fixture detail supports rich includes such as:
  - `balls`
  - `runs`
  - `batting`
  - `bowling`
  - `scoreboards`
  - `lineup`
- this supports a two-level polling model:
  - light live discovery from `/livescores`
  - targeted detail refresh for only the active matches that actually matter

Source checked:
- SportMonks Cricket API docs: `GET Fixture by ID`
- endpoint shown as `https://cricket.sportmonks.com/api/v2.0/fixtures/{ID}`
- docs also list `Livescores` as the live endpoint family

## 3-Phase Upgrade Plan

### Phase 1: Batch Discovery Layer
Build a dedicated SportsMonks cricket discovery service that polls live state in batches, not per-match.

What to do:
- keep `/livescores` as the batch live discovery source
- poll it on a bounded schedule for active cricket feeds
- normalize each returned fixture into a compact live index:
  - fixture id
  - live flag/status
  - league/season ids
  - score summary
  - innings/over/ball markers
  - last activity timestamp
- store this live index in Redis or ETS with a short TTL
- stop relying on repeated broad competition fetches to answer per-match live freshness questions

Output of Phase 1:
- one lightweight batch pass tells the system which matches are live and changing
- the rest of the stack can decide which of those matches deserve deeper refresh

### Phase 2: Targeted Detail Refresh Layer
After batch discovery, only fetch full detail for matches that are operationally important.

Important matches include:
- mapped matches with live odds on screen
- matches currently being repriced by LangGraph
- matches with stale or degraded provider state
- matches in the user-visible sportsbook live board
- matches near critical state transitions:
  - wicket
  - over end
  - innings break
  - chase pressure spike

What to do:
- build a targeted detail fetcher around `/fixtures/{id}`
- use richer includes only for selected matches
- add per-match TTL classes:
  - hot live: 2 to 5 seconds
  - normal live: 8 to 15 seconds
  - low-interest live: 20 to 30 seconds
  - innings break / paused: 30 to 60 seconds
- never refetch every live match at the same full-detail cadence

Output of Phase 2:
- the engine gets rich ball-by-ball context where it matters
- the platform avoids expensive full-detail refreshes for every live match

### Phase 3: Priority Scheduler and Backpressure Control
Add a scheduler that decides which matches get detail refresh capacity when the platform is under load.

What to do:
- score each live match by priority:
  - user-visible board open
  - active betting volume
  - mapped 1xBet source available
  - AI repricing active
  - stale/degraded state
  - recent important cricket event
- refresh highest-priority matches first
- add backpressure limits:
  - maximum full-detail refreshes per cycle
  - maximum refresh concurrency
  - cooldown after repeated unchanged snapshots
- add per-match activity memory:
  - unchanged count
  - last detailed refresh
  - last material event seen
  - last publish timestamp

Output of Phase 3:
- load stays predictable
- more leagues can be enabled without blindly multiplying request volume
- live user-facing matches stay fresh first

## Required Engineering Changes

### Backend
- add a cricket live discovery worker for SportsMonks batch polling
- add a targeted fixture-detail worker keyed by fixture ids
- add Redis/ETS live index cache for discovery snapshots
- update `MatchFetcher` to stop treating all live sync work as one generic loop
- add priority scheduling inputs from:
  - Matchmaker mapping state
  - live board visibility
  - stale-provider status
  - AI repricing activity

### AI / Repricing
- keep LangGraph per-match context, but feed it from the detail refresh layer
- persist per-match polling metadata alongside existing match memory
- use unchanged snapshot detection so the AI graph is not re-run on no-op updates

### UI / Ops
- add admin visibility for SportsMonks polling state:
  - last batch discovery run
  - live fixtures discovered
  - detail refresh queue depth
  - top throttled matches
- expose per-match freshness source:
  - discovery fresh
  - detail fresh
  - degraded
  - awaiting targeted refresh

## Edge Cases To Handle
- live score updates with no ball feed change
- innings break with markets still published
- temporary SportMonks lag while 1xBet source odds remain fresh
- abandoned/no-result transitions
- repeated identical snapshots causing useless repricing churn
- sudden spike in concurrent live matches across many leagues
- matches visible to users but not currently receiving full-detail refresh

## Success Criteria
The upgrade is successful when:
- broad SportsMonks request volume drops materially
- more leagues can be enabled without UI degradation
- user-visible live cricket matches remain fresh
- AI repricing only runs on meaningful state changes
- no live match board empties because the polling model starved the detail path
- the platform can distinguish:
  - discovery freshness
  - detail freshness
  - odds freshness
  - publish freshness

## Recommended Implementation Order
1. Phase 1 batch discovery worker and live index cache
2. Phase 2 targeted detail refresh for mapped/user-visible matches
3. Phase 3 priority scheduler and backpressure controls

## Recommendation
Do not replace the current SportsMonks path in one step.
Ship this as a layered migration:
- keep current live polling as fallback
- enable batch discovery first
- route only selected cricket feeds through the new targeted detail path
- compare request volume and live board stability before full cutover
